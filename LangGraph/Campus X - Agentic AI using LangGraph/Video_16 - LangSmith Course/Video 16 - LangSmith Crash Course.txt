1. Let's say you have build a complex LangGraph LLM WOrkflow and suddenly you see a spike in latency or cost and now you need to figure out which node is causing this problem. For this you require observability and that can be achieved using LangSmith.

2. Observability:
	a. Observability is the ability to understand a system's internal state by examining its external outputs, like logs, metrics, and traces.

	b. It allows you to diagnose issues, understand performance, and improve reliability by analyzing data generated by the system.

	c. Essentially, it's about being able to answer "why" something is happening within a system, even if you didn't anticipate the problem.
	
3. LangSmith:
	a. LangSmith is a unified observability & evaluation platform where teams can debug, test, and monitor AI app performance.
	
4. What does LangSmith Trace?
	a. Input and Output
	b. All the intermediate steps
	c. Latency
	d. Token usage
	e. Cost
	f. Error
	g. Tags
	h. Metadata
	i. Feedback

6. Core Concepts:
	a. Project (Container)
		i. Represents a single application, allowing you to group traces together for monitoring, debugging, and evaluation (e.g., "production-chatbot", "testing-rag-app").
	
	b. Trace (Execution Log):
		i. A tree structure representing one complete interaction. It displays the flow of data, latency, and costs from start to finish.
		
	c. Run (Atomic Action): 
		i. Individual, nested steps within a trace (e.g., a specific OpenAI call, a vector DB retrieval). A run is a span representing a single unit of work or operation within your LLM application. This could be anything from a single call to an LLM or chain, to a prompt formatting call, to a runnable lambda invocation. . Runs can be categorized into types:
			- LLM: Calls to a model, often showing prompts and generated answers.
			- Retriever: Documents fetched from data sources.
			- Tool: Actions triggered by agents.
			- Chain: Sequences of operations.
			
	d. A trace records the sequence of steps your application takes—from receiving an input, through intermediate processing, to producing a final output. Each step within a trace is represented by a run. Multiple traces are grouped together within a project, and traces from multi-turn conversations can be linked together as a thread.
